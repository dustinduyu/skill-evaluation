# Skill / Agent 评测流程

写报告前，先用这份文件设计评测流程。本文的核心思路来自 skill 评测中的“是否能被调用、是否优于无 skill 基线”视角，以及 agent 质量评测中的分层观测、trace / logs / metrics 证据化方法。[1][2]

## 1. 适用对象

这套工作法不只适用于单一 skill，也适用于由多个能力串联而成的复杂 skill，以及具备自主执行能力的 agent。

| 对象 | 说明 | 评测使用方式 |
| --- | --- | --- |
| 单一 skill | 只完成一个相对清晰的能力，例如检索、改写、评分、生成报告 | 可以轻量使用本工作法，重点关注输入输出是否稳定、结果是否符合预期 |
| 多环节 / 复杂 skill | 由多个步骤或多个 skill 串联完成任务，例如来源获取、内容评分、结果整理、分发 | 建议完整使用本工作法，重点关注各环节是否可观测、责任边界是否清楚、链路是否稳定 |
| Agent | 具备目标理解、任务拆解、工具选择、状态管理、失败恢复、多轮执行等能力的工作流 | 应完整使用本工作法，并重点关注任务成功率、工具使用质量、长期稳定性和失败恢复能力 |

使用时不需要先严格判断对象属于哪一类。上表只是帮助确定评测的复杂度和投入程度：对象越简单，评测可以越轻量；对象越复杂，越应该完整执行后续流程。

## 2. 理解场景与评测对象

需要阅读：

- 用户需求、业务场景、使用对象和成功标准
- `SKILL.md`
- `references/`、`scripts/`、`assets/` 下被引用的文件
- skill 实际调用的本地代码路径
- 已有日志、trace、metrics、数据集、历史报告
- 用户提供的工作法或参考样例

需要记录：

| 字段 | 记录内容 |
| --- | --- |
| 使用场景 | 这个 skill 在什么业务场景中被使用，谁会使用 |
| 成功标准 | 什么结果算有用，哪些取舍需要人工确认 |
| 链路复杂度 | 单步闭合、多环节依赖、自主决策/失败回流等维度分别是否存在 |
| 核心任务 | 这个对象主要负责什么 |
| 输入 | prompt、文件、数据集、API、上游产物 |
| 输出 | 结构化结果、报告、文件、动作、判断 |
| 依赖 | API、key、CLI、模型、仓库、工具 |
| 边界 | 这个 skill 不负责什么 |

不需要把评测对象硬分成固定类型。下面三个维度不是互斥选项，可以同时成立；先逐项勾选，再决定评测深度：

| 复杂度维度 | 判断问题 | 成立时需要关注 |
| --- | --- | --- |
| 功能闭合度 | 输入输出是否能单步闭合 | 输入输出稳定性、结果正确性、边界条件 |
| 链路依赖度 | 是否由多个相互依赖环节组成 | 环节可观测、责任边界、链路稳定性 |
| 自主决策与回流 | 是否存在多步决策、工具选择、状态传递或失败回流 | 任务成功率、评判策略、失败恢复、长期稳定性 |

报告里说明每个维度是否成立，以及因此增加了哪些评测重点。无论复杂度如何，后续报告仍固定使用三层：运行框架层、能力链路层、业务效果层。

## 3. 使用固定三层

始终使用这三层。三层不是按“代码模块”切分，而是按评测时要定位的问题切分：先看能不能被正确唤起并稳定运行，再看中间能力链路是否按预期工作，最后看结果是否满足业务目标。这样可以把运行问题、过程问题和效果问题分开定位，同时保留 trace / logs / metrics 能支撑的证据链。[2]

| 层级 | 具体看什么 | 不看什么 | 常见指标 |
| --- | --- | --- | --- |
| 运行框架层 | skill 能否被正确唤起；入口命令/API/工具是否可用；权限、凭证、环境变量、路径、依赖是否满足；日志、trace、metrics、成本、耗时、错误、重试、运行状态是否可记录 | 不判断最终内容好不好；不评价某一步语义判断是否聪明 | 唤起准确率、运行成功率、错误率、超时率、成本、耗时、重试次数、日志完整性、状态可追踪性 |
| 能力链路层 | 输入如何被拆解、转换和传递；数据准备、检索、抓取、解析、清洗、生成、评分、排序、工具调用、中间产物和 Schema 是否正确 | 不把 API 掉线、权限失败归到这里；不直接用最终业务喜好替代过程判断 | 步骤成功率、Schema 完整性、字段覆盖率、召回覆盖、转换一致性、工具选择正确率、中间产物可用率 |
| 业务效果层 | 最终输出是否完成用户或业务目标；质量、相关性、准确性、完整性、可读性、安全边界、误判、漏判、人工复核结果；是否优于无 skill 基线 | 不追究底层权限、网络和环境问题；不把过程看起来合理等同于最终有效 | 准确率、相关性、误判率、漏判率、有效结果数、人工通过率、用户价值、相对基线提升 |

每个观察点都按需启用。普通单步 skill 只保留必要指标；复杂工作流可以在同一张三层表里补充额外观察点，不新增第四层。

## 4. 定义目标和指标

每次评测都需要覆盖：

1. **运行目标**：能否稳定运行。
2. **能力目标**：内部链路能否产出可用中间结果。
3. **业务目标**：是否解决用户或业务问题。
4. **唤起目标**：agent 能否正确选择这个 skill。[1]
5. **必要性目标**：相比无 skill 基线，是否有质量或稳定性收益。[1]

每个指标都要定义：

| 字段 | 含义 |
| --- | --- |
| 指标名 | 简洁中文名称 |
| 所属层级 | 固定三层之一 |
| 计算规则 | 明确公式或判断口径 |
| 数据来源 | 日志、JSON、trace、模型输出、人工标签 |
| 评判方式 | 代码、规则、模型、人工或混合 |
| 结果 | 实测值或 `待补测` |

同时检查文档与实现是否一致：

| 检查项 | 对比内容 |
| --- | --- |
| 职责覆盖 | `SKILL.md` 中声明的职责 vs 实际命令/脚本是否覆盖 |
| 命令覆盖 | 常用命令 vs 已声明的全部职责 |
| 副作用覆盖 | 写回、归档、发送等动作 vs 安全规则、身份和权限要求 |
| 可移植性 | 绝对本地路径、硬编码凭证、本地 CLI、命名兼容性 |

## 5. 评测设计确认

在进入正式测试前，先形成一版评测设计草案，并让用户确认。草案确认后再执行会产生明显成本、外部副作用或在线写入的动作。

确认内容：

| 模块 | 需要确认什么 |
| --- | --- |
| 三层拆解 | 运行框架层、能力链路层、业务效果层是否覆盖了关键职责 |
| 目标与指标 | 评什么、为什么评、每个指标如何计算 |
| 数据方案 | 数据来源、样本筛选、数据集拆分、reference output 或人工金标来源 |
| 评判方式 | 哪些用代码评判、规则评判、模型评判、人工评判或混合评判 |
| 数据记录 | 运行数据和评判结果记录到哪里，字段是否够后续复盘 |
| 测试流程 | 测试数据集构建、执行步骤、是否需要 multi-agent / sub-agent 协同、评测结果数据存档方式 |
| 输出形态 | 报告面向谁、首页状态总览展示哪些检查项 |

确认前可以做：

- 阅读文件、日志和历史产物。
- 设计指标、数据表和测试计划。
- 运行低风险、可逆的小样本验证。

确认后再做：

- 批量运行测试。
- 调用外部 API 产生明显成本。
- 更新在线文档或数据库。
- 生成最终报告并沉淀为正式 eval 资产。

测试流程中的协同编排：

复杂评测可以在测试流程内部设计 multi-agent / sub-agent 协同，但只有当用户明确要求或当前环境允许时才实际启用。协同的主要目的不是多开任务本身，而是把复杂 skill / agent 的不同评测观察面隔离开，让每一部分独立产出证据，减少上下文互相影响。

适合拆分的评测观察面：

| 观察面 | 适合交给谁 | 产出 |
| --- | --- | --- |
| 运行框架评测 | 运行评测子任务 | 入口、权限、日志、成本、错误、状态记录 |
| 能力链路评测 | 链路评测子任务 | 规划、工具选择、中间产物、Schema、状态交接 |
| 业务效果评测 | 效果评测子任务 | 最终质量、误判漏判、人工复核、业务价值 |
| 基线对照评测 | 基线评测子任务 | 无 skill 基线结果、对照差异 |
| 分歧复核 | 复核子任务或人工 | 分歧样本表、复核口径 |

编排原则：

- 主 agent 保留评测设计、用户确认、最终结论和线上写入的控制权。
- 子任务必须有清晰边界和产出路径。
- 涉及同一文件写入时避免并行修改。
- 子任务结果需要进入统一的数据记录表，不只停留在聊天总结里。

## 6. 数据集拆分

当目标会持续迭代时，使用三类数据集。这里借鉴传统机器学习中训练集、验证集、测试集的职责分离思想：用于调规则的数据、用于日常防退化的数据、用于阶段验收的数据应尽量分开，避免只对当前样本过拟合。[3][4]

拆分前先完成测试数据集构建：确定数据来源，清洗和标准化字段，补齐 reference output 或人工金标，并保留原始数据路径。合成样本可以用于补齐稀有场景、负例或提升泛化覆盖，但必须标记来源，不替代真实数据。

| 集合 | 用途 | 对应思想 |
| --- | --- | --- |
| 优化集 | 调规则、prompt、阈值和行为 | training / development |
| 回归集 | 每次修改后重跑，防止退化 | validation / regression |
| 保留集 | 阶段性验收，谨慎使用 | test / holdout |

按 skill 阶段建议比例：

| 阶段 | 优化集 | 回归集 | 保留集 |
| --- | --- | --- | --- |
| 起步阶段 | 50%-60% | 25%-35% | 10%-15% |
| 成长期 | 35%-45% | 35%-45% | 15%-20% |
| 稳定应用阶段 | 20%-30% | 50%-60% | 15%-25% |
| 高风险上线前 | 15%-25% | 50%-60% | 20%-30% |

拆分规则：

- 当样本类型差异明显时，使用分层拆分。
- 边界样本和已确认失败样本进入回归集，后续每次改动都复测。
- 保留集的样本和标签应保持稳定，不要反复拿它调规则。
- 如果样本池很小，明确说明限制，并加强定性失败分析。

## 7. 评测数据记录

运行脚本产生的数据必须真实保存。前期可以用表文件，后续可以升级为 SQLite、LangSmith dataset / experiment、飞书多维表格或正式数据库。

建议目录：

| 目录 | 内容 |
| --- | --- |
| `eval_runs/` | 每次运行的 run metadata、命令、耗时、成本、环境信息 |
| `datasets/` | 全量样本、优化集、回归集、保留集、reference output |
| `results/` | 样本级输出、评判结果、分歧样本、汇总指标 |
| `artifacts/` | 截图、日志、trace、报告附件 |

最小表结构：

| 字段 | 说明 |
| --- | --- |
| `run_id` | 本次评测运行 ID |
| `case_id` | 样本 ID |
| `dataset_split` | optimization / regression / holdout / ad_hoc |
| `input_ref` | 输入文本或输入文件路径 |
| `expected_ref` | reference output、人工金标或预期文件路径 |
| `actual_output_ref` | 实际输出文件路径或摘要 |
| `layer` | 运行框架层 / 能力链路层 / 业务效果层 |
| `metric_key` | 指标名 |
| `judge_type` | code / rule / llm / human / mixed |
| `score` | 数值分或分类标签 |
| `pass_fail` | pass / fail / partial / pending |
| `error` | 错误信息或空 |
| `duration_ms` | 样本或步骤耗时 |
| `cost` | token、API 或其他成本 |
| `artifact_path` | 日志、trace、截图、输出文件路径 |
| `review_status` | unreviewed / reviewed / needs_human |
| `created_at` | 记录时间 |

每次运行后至少保存：

- run 级记录：命令、环境、输入数据版本、输出目录、总耗时、总成本。
- case 级记录：每条样本的输入、输出、评判结果和错误。
- metric 级记录：每个指标的聚合结果和计算口径。

## 8. 闭环沉淀

评测不是写完报告就结束。只有当发现被沉淀成可复用资产，后续修改才能防退化、可复盘、可继续优化。

| 沉淀内容 | 沉淀位置 |
| --- | --- |
| 全量样本 | dataset 目录 |
| 优化集 | `optimization_set.*` |
| 回归集 | `regression_set.*` |
| 保留集 | `holdout_set.*` |
| 分歧样本 | 报告表格和数据集 |
| 评判口径 | 报告或 skill references |
| skill 修复项 | 优化方案、issue 或 PR |
| 唤起提示词 | invocation eval set |
| 无 skill 基线 | baseline artifact |
| 运行数据表 | `eval_runs/`、`results/` 或指定数据库 |

## 9. 静态评测模式

当目标 skill 没有安全的可执行入口，或用户只提供 skill 仓库但没有授权外部动作时，使用静态评测模式。

静态评测也要产出完整报告，但不能假装已经测过真实输出质量。

| 步骤 | 具体动作 | 结果标签 |
| --- | --- | --- |
| metadata 校验 | 校验 `SKILL.md` frontmatter、name、description | `已测` |
| 文件盘点 | 阅读 `SKILL.md`、references、scripts、README、agents metadata | `已测` |
| 流程覆盖 | 检查是否说明输入、输出、边界、流程、质量检查 | `已测` |
| 依赖与安全检查 | 识别外部工具、凭证、在线写回、破坏性动作 | `已测` |
| 唤起设计 | 设计正例、负例、相似 skill 干扰提示词 | 未跑 router 日志时标 `待补测` |
| 真实输出质量 | 用真实任务运行 skill 并评判输出 | 未实际运行时标 `待补测` |
| 无 skill 基线 | 和通用模型临场处理对比 | 未实际运行时标 `待补测` |

静态评测的 `测试结果` 部分应包含：

- `本轮执行概览`
- `静态结构检查`
- `可唤起性与存在必要性检查`
- `发现的问题`
- 可选的 `待补测计划`
